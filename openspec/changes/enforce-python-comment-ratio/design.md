## 上下文

`vllm_bench_platform/` 是平台核心源码目录。当前代码处于 MVP 初始实现阶段，已经包含配置 schema、后端 ConfigMap/PVC/Job builder 和 submit-job 编排。用户要求该目录源码注释量不得低于 40%，因此需要用自动化方式固化检查口径。

## 目标 / 非目标

**目标：**

- 统计 `vllm_bench_platform/**/*.py` 的整体注释率。
- 注释率低于 40% 时让测试失败。
- 用中文注释解释领域意图、Kubernetes 资源设计、OpenSpec 约束和失败处理。
- 将该规范写入 AGENTS 和 OpenSpec config，后续实现继续遵守。

**非目标：**

- 不改变业务行为。
- 不为了凑比例添加无意义逐行注释。
- 不要求第三方依赖。

## 技术决策

### 决策：用标准库 tokenization 统计注释率

理由：Python 标准库 `tokenize` 可以识别 `#` 注释；结合 AST 可以识别模块、类、函数文档字符串，避免引入新依赖。

备选方案：用简单字符串匹配。该方案容易误判字符串里的 `#`，不采用。

### 决策：按目录整体统计

理由：当前用户要求是“所有的 vllm_bench_platform 里面”，整体统计能避免小型 `__init__.py` 因文件短而扰动结果。

备选方案：逐文件要求 40%。这会迫使短文件写无意义注释，不采用。

## 风险 / 权衡

- 40% 注释率较高 -> 注释必须解释意图和约束，避免机械复述代码。
- 文档字符串统计口径可能与人工直觉不同 -> 用脚本固定规则，测试使用同一口径。

## 迁移计划

1. 新增注释率统计工具。
2. 新增测试，先观察当前注释率不达标。
3. 为 `vllm_bench_platform/` 源码补充有意义的中文注释和文档字符串。
4. 更新 AGENTS 和 OpenSpec config。
5. 运行 OpenSpec strict 校验和全量单元测试。
