## 原因

`vllm_bench_platform/` 是后续平台核心代码目录，需要把国产卡、Kubernetes、压测矩阵和失败处理等业务约束写清楚。用户明确要求该目录内源码注释量不得低于 40%，因此需要形成可验证的代码质量规范。

## 背景

当前 MVP 已开始实现，`vllm_bench_platform/` 下已经包含 schema、后端资源 builder 和 submit-job 编排代码。为了避免后续实现只堆代码、不沉淀设计意图，需要将注释率要求纳入 OpenSpec 和自动化测试。

## 目标

- 要求 `vllm_bench_platform/` 下 Python 源码整体注释率不低于 40%。
- 注释率统计包含 `#` 注释行和文档字符串行，按非空源码行作为分母。
- 注释必须解释业务约束、OpenSpec 追溯、Kubernetes 资源意图、失败处理原因或维护注意点。
- 提供自动化检查，防止后续代码把注释率降到 40% 以下。

## 非目标

- 不要求每个函数都写长注释。
- 不要求无意义逐行翻译代码。
- 不改变 MVP 架构、接口和运行逻辑。

## 用户故事

- 作为项目维护者，我希望关键代码保持足够注释，便于后续理解国产卡压测平台的业务约束。
- 作为实现者，我希望有自动化检查告诉我注释率是否达标，而不是靠人工估算。

## 约束条件

- 注释率目标只作用于 `vllm_bench_platform/**/*.py`。
- 空行不计入分母。
- 文档字符串行和 `#` 开头注释行计入注释行。
- 注释不得只是重复代码字面行为，必须说明领域意图、约束来源或维护注意事项。

## 变更内容

- 新增 `code-quality` capability。
- 新增注释率检查脚本。
- 新增单元测试，要求 `vllm_bench_platform/` 注释率不低于 40%。
- 为现有 `vllm_bench_platform/` Python 源码补充中文注释和文档字符串。

## 能力范围

### 新增能力

- `code-quality`：定义并验证 `vllm_bench_platform/` Python 源码注释率不低于 40%。

### 修改能力

- 无。

## 验收标准

- OpenSpec strict 校验通过。
- 自动化测试能统计 `vllm_bench_platform/**/*.py` 的注释率。
- 当前 `vllm_bench_platform/` 注释率不低于 40%。
- 现有单元测试继续通过。

## 风险点

- 40% 注释率较高，可能诱导无意义注释；通过要求注释说明业务意图和维护约束来缓解。
- 注释率统计可能与人工理解有差异；通过固定脚本口径保证一致性。

## 影响范围

- `vllm_bench_platform/**/*.py`
- `tests/`
- `tools/`
- `.codex/AGENTS.md`
- `openspec/config.yaml`
